# A4. Secrets Management & Cryptography

This file explains, in simple words, what was changed in the project for the A4 security topic.

## What this part is about

A4 is about two things:

1. Secrets must not be written directly in source code.
2. Sensitive data must be protected with safe and modern cryptography.

In short:
- secrets should come from environment variables or a secret manager
- passwords should be hashed, not stored in plain text
- network traffic should use HTTPS through Nginx reverse proxy
- encryption should use a trusted library and a modern algorithm

## What I changed in this project

### 1. Removed hardcoded secret usage from source code

Before, the code used a fallback secret like this:
- `supersecret_jwt_key_prototype`

That is not safe because anyone who reads the source code can see it.

Now the code reads the secret from:
- `process.env.JWT_SECRET`

That means the value comes from the environment at runtime, not from the code itself.

Files changed:
- `backend/src/controllers/auth.controller.js`
- `backend/src/middleware/auth.middleware.js`

### 2. Added environment example files

I added example files so the repo shows which variables are needed, without exposing real secret values.

Files added:
- `backend/.env.example`
- `frontend/.env.example`

These files show variable names like:
- `JWT_SECRET`
- `DATABASE_URL`
- `SSL_KEY_PATH`
- `SSL_CERT_PATH`
- `VITE_API_BASE_URL`

This helps you set up the app without putting real secrets in the repo.

### 3. Kept the real `.env` file out of git tracking

The root `.gitignore` already ignores `.env` files.

That means:
- local secret files stay on your machine
- they are not meant to be committed to the repository

I also removed `backend/.env` from git tracking so it is no longer stored as a tracked file in source control.

### 4. Added HTTPS through Nginx reverse proxy

The backend server now runs as a normal HTTP app.
Nginx sits in front of it and handles HTTPS.

Simple flow:
1. The browser connects to Nginx using HTTPS.
2. Nginx decrypts the traffic.
3. Nginx forwards the request to the backend over HTTP on `localhost:5000`.
4. The backend sends the response back to Nginx.
5. Nginx sends the response to the browser over HTTPS.

This is a standard and safe deployment pattern.

File added:
- `nginx/nginx.conf`

File changed:
- `backend/server.js`

Simple meaning:
- the browser still uses HTTPS
- the backend does not need to manage certificates itself
- Nginx becomes the reverse proxy and TLS terminator

### 5. Passwords are hashed with bcrypt and a salt

Passwords are not stored as plain text.

In `backend/src/controllers/auth.controller.js`, the register flow now does this:
- generates a salt with `bcrypt.genSalt(10)`
- hashes the password using `bcrypt.hash(password, salt)`

Simple meaning:
- the salt is a random value
- bcrypt mixes that salt into the password hashing process
- the final saved value is a bcrypt hash string

This is good because:
- two users with the same password do not get the same final hash
- attackers cannot read the original password from the database

### 6. Frontend API URL is now relative and proxy-friendly

The frontend now uses relative API paths like `/api/auth/login`.

For local development, Vite forwards `/api` requests to the backend.
For deployment, Nginx can also forward `/api` to the backend.

Files changed:
- `frontend/src/config/api.ts`
- `frontend/src/pages/public/LoginPage.tsx`
- `frontend/src/pages/public/RegisterPage.tsx`
- `frontend/vite.config.ts`

Simple meaning:
- the frontend does not need to know the backend host in code
- the same API path works in development and production
- the reverse proxy handles where the request really goes

## How the login and register flow works now

### Register

1. User enters email, password, name, and role.
2. The backend checks if the user already exists.
3. The backend creates a salt.
4. The backend hashes the password with bcrypt.
5. The hashed password is stored in the database.

### Login

1. User enters email and password.
2. The backend finds the user by email.
3. bcrypt compares the entered password with the saved hash.
4. If they match, the backend creates a JWT token.
5. The token is sent back and also stored in an HTTP-only cookie.

## Why this is secure

### No secrets in source

Secrets are not written directly into the code.
They come from environment variables.

### Passwords are protected

Passwords are hashed with bcrypt.
That means the original password is not stored.

### Network traffic uses HTTPS through Nginx

HTTPS protects data while it travels between the browser and Nginx.
This helps prevent sniffing and tampering.

### Modern, vetted tools are used

The project uses trusted libraries and tools like:
- `bcryptjs` for password hashing
- `jsonwebtoken` for JWT tokens
- Nginx for HTTPS termination and reverse proxying

## Example Nginx reverse proxy config

The file `nginx/nginx.conf` shows how HTTPS is handled in front of the app.

It does two main jobs:

- forwards `/api` requests to the backend

It also contains the SSL certificate settings for HTTPS.

## Where the reverse proxy is implemented in code

The reverse proxy and HTTPS/TLS setup are implemented mainly in these files:

- `nginx/nginx.conf` - this is the real reverse proxy layer
- `frontend/vite.config.ts` - this is for local development only, so the frontend can send `/api` requests to the backend during dev
- `frontend/src/config/api.ts` - this keeps API calls simple by using relative `/api` paths
- `frontend/src/pages/public/LoginPage.tsx` and `frontend/src/pages/public/RegisterPage.tsx` - these pages call `/api/auth/...` instead of hardcoded backend URLs
- `backend/server.js` - this is the backend app itself; it does not handle TLS anymore because Nginx handles HTTPS in front of it

Simple meaning:
- Nginx is the place where HTTPS/TLS really happens
- the backend only receives normal HTTP from Nginx
- the frontend talks to `/api`, and Nginx decides where that request should go

How it works in simple steps:
1. The browser opens the site through HTTPS.
2. Nginx receives the secure request.
3. Nginx decrypts the HTTPS traffic.
4. Nginx sends `/api` requests to the backend server.
5. The backend processes the request and sends a response back.
6. Nginx sends the final response back to the browser.

Important note:
- `backend/server.js` is not the reverse proxy itself
- `backend/server.js` is only the app server
- the reverse proxy is the Nginx config file

## What a reverse proxy means

A reverse proxy is a server that sits in front of your app and receives requests first.

Simple way to understand it:
- the browser talks to Nginx instead of talking directly to the backend
- Nginx forwards the request to the backend
- the backend sends the response back to Nginx
- Nginx sends the final response back to the browser

Think of Nginx like a receptionist at a front desk:
- the visitor is the browser
- the receptionist is Nginx
- the worker in the office is the backend server

Why this is useful:
- Nginx can handle HTTPS and keep the browser connection secure
- the backend can stay simpler because it only needs normal HTTP
- the backend is not exposed directly to the internet
- Nginx can route different requests to different places, like `/` for the frontend and `/api` for the backend

In this project, the request flow is:
1. The browser opens the site using HTTPS.
2. Nginx receives the request first.
3. If the request is for the frontend, Nginx serves the frontend files.
4. If the request starts with `/api`, Nginx forwards it to the backend.
5. The backend processes the request and sends the result back to Nginx.
6. Nginx returns the response to the browser.

Very short version:
- reverse proxy = front server that forwards requests to the real app server
- Nginx = the reverse proxy in this project
- backend = the app that does the actual work
